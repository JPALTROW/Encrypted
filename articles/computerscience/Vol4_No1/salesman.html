The Traveling Salesman Problem

Ben Pan

Introduction

Imagine you are a salesman with the task of promoting a product in various cities. There is a road between every pair of cities, and you are given the length of each road.

Your goal is to construct the shortest possible route visiting each city exactly once and returning to the origin. We will call a route that visits every vertex (city) exactly once and returns to the origin a Hamiltonian Cycle.

We can model the Traveling Salesman Problem (TSP) as a graph theory problem:

Input:

  • A complete graph with weighted edges G‚ÄÑ=‚ÄÑ(V,E)

    • A complete graph is a graph in which every pair of vertices is connected by an edge.

    • V is a set of vertices which represent the cities.

    • E is a set of edges which represent the roads.

  • The weight of each edge, representing the length of roads.

Output:

The length of the shortest Hamiltonian Cycle of G.

What is NP-hardness?

In theoretical computer science, an NP-hard problem is a problem that currently does not have a solution in polynomial time. Informally speaking, NP-hard problems cannot be solved in a reasonable amount of time.

For example, consider the Traveling Salesman Problem. One simple algorithm might be the following:

  1. Generate all n! permutations of vertices, where n is the number of vertices. For example, in a graph with 4 vertices, 2 possible paths are 4‚ÄÑ‚Üí‚ÄÑ2‚ÄÑ‚Üí‚ÄÑ1‚ÄÑ‚Üí‚ÄÑ3 and 2‚ÄÑ‚Üí‚ÄÑ1‚ÄÑ‚Üí‚ÄÑ3‚ÄÑ‚Üí‚ÄÑ4.

  2. Notice that each permutation of vertices is a valid route. In the examples above, the corresponding routes are 4‚ÄÑ‚Üí‚ÄÑ2‚ÄÑ‚Üí‚ÄÑ1‚ÄÑ‚Üí‚ÄÑ3‚ÄÑ‚Üí‚ÄÑ4 and 2‚ÄÑ‚Üí‚ÄÑ1‚ÄÑ‚Üí‚ÄÑ3‚ÄÑ‚Üí‚ÄÑ4‚ÄÑ‚Üí‚ÄÑ2. We can just append the first vertex to the end of the permutation to generate the route.

  3. Iterate through each permutation, calculate the ‚Äúcost", and keep track of the minimum ‚Äúcost".

  4. Return the route with minimum cost.

The complexity of this algorithm is ùí™(n!), which means that the computer must run through about n! calculations.

Let us assume each ‚Äústep" takes 0.5 nanoseconds. For a graph with n‚ÄÑ=‚ÄÑ5 vertices, this might only take the computer 5!‚ÄÑ=‚ÄÑ120 ‚Äústeps". The algorithm will finish running in under a second!

However, consider a graph with n‚ÄÑ=‚ÄÑ1000 vertices. This algorithm will take the computer 1000! steps, which is an astronomical number. Even the best supercomputers in the world cannot finish running the algorithm before you die!

For decades, people have been attempting to solve the Traveling Salesman problem, but no one has ever come up with an algorithm that can be run in a reasonable amount of time. In the following section, I will provide a proof that TSP is an NP-hard problem. In simpler words, solving TSP is impossible under current circumstances.

Proof that TSP is NP-hard

We will prove that the Traveling Salesman Problem is NP-hard by proving that it is as hard as the NP-hard Hamiltonian Cycle (HC) problem. The Hamiltonian Cycle Problem determines if there exists a HC in the graph. In other words, we will show that if we have a polynomial time algorithm to solve TSP, we can also solve the HC problem in polynomial time.

Take an arbitrary instance of the HC Problem with a graph G‚ÄÑ=‚ÄÑ(V,E). We will construct a complete graph G‚Ä≤ as follows: For every pair of vertices (vi,vj), if there does not exist an edge between them, then we will add in an edge of weight 1. Otherwise, if there is an edge between them, then we will add a weight of 0 to that edge. This is to make the graph complete for the Traveling Salesman Problem.

Since we now have a complete graph G‚Ä≤, we have successfully reduced the HC Problem to the Traveling Salesman Problem. The tour that satisfies the requirements of TSP is a cycle with n vertices and thus n edges. The ‚Äúcheapest" tour is made up of n edges of weight 0. All edges of weight 0 belong to E, the original set of edges in the HC problem, and since the tour visits every vertex and is a cycle, then we have found a Hamiltonian Cycle in the original graph G. Thus, we have reduced TSP to the HC Problem.

In conclusion, any algorithm that can solve TSP in polynomial time can also solve HC in polynomial time, but no such algorithm exists because HC is NP-hard, so TSP must be NP-hard as well.

MST-based Heuristic

Claim: There exists a heuristic based on minimum spanning trees that will yield a solution at most 2 times the optimal solution and will run in polynomial time, specifically O(E+VlogV).

The heuristic is surprisingly simple with the following steps:

  1. Construct a minimum spanning tree (MST) of the graph using Kruskal‚Äôs or Prim‚Äôs Algorithm.

  2. Traverse the MST, taking shortcuts if necessary.

Minimum Spanning Tree: A subgraph of a connected, undirected graph with weighted edges such that

  1. It is a tree.

  2. It connects all vertices of the graph.

  3. The sum of the edge weights is minimum.
