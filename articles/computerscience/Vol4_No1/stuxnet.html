Stuxnet

Daniel Lyalin

Introduction

Early in July 2009, Gholam Reza Aghazedah, then head of the Atomic Energy Organization of Iran, resigned following a “serious nuclear accident" that had forced the removal of 900 to 1,000 centrifuges used for uranium enrichment at Iran’s Natanz facilities. In fact, the computer virus that took out these centrifuges was not "accidental," but the result of a coherent, long-term cyberattack on the Iranian nuclear program, likely the result of joint NSA-CIA-Mossad operations. Exploiting four (!) zero-day vulnerabilities, the Stuxnet virus, as it would come to be called, is considered one of the most advanced pieces of malware written, with the flaws it exposed still salient cybersecurity issues — Windows released a patch in May 2020 addressing zero-day #4, the spooler subsystem bug that Stuxnet took advantage of.

Stuxnet’s Spread

Unlike most modern viruses, Stuxnet initially spread not via email attachments or similar Internet mechanisms, but via USB drives, a more reliable method of closing the air-gap and infecting devices. Having installed itself onto a Windows-running machine, Stuxnet exploited four Windows zero-days (explained below) to escalate privilege, or increase the security clearance it had, until it had a rootkit installed; rootkits are programs loaded before the OS is activated with a root security clearance, usually used by malware to prevent detection and interact with low-level computer components. Stuxnet, being a virus designed to target centrifuge controllers, scanned the Windows environment of infected machines to detect whether those machines were running Siemens’ Step 7 SCADA system, the control software used to interact with centrifuges in enrichment facilities. If the SCADA system was found, Stuxnet installed a wrapper on the driver DLL (a file containing instructions for a program to execute) for the communications between the SCADA and the logic controllers interacting with the centrifuges, launching a Man-in-the-Middle attack. However, Stuxnet was designed not to destroy the Natanz facility, but to control and deteriorate it; to achieve this goal effectively, it had to avoid detection. Because of this, Stuxnet, after installation, spent approximately 27 days monitoring and recording SCADA activity. This was intended to make sure that Stuxnet had infected the right machine, as well as to fake legitimate data for the SCADA when it began messing with the centrifuges, which it only did for 21 seconds at a time in between its 27-day monitoring periods. During these brief attacks, Stuxnet either sped the centrifuges up to their maximum rotor speed to cause maximum fatigue, or slowed them down dramatically; while not directly damaging the centrifuges, the induced fatigue wore them down rapidly and eventually caused them to break during normal operations. Regardless of whether Stuxnet detected the Step 7 software or not, though, it continued spreading by exploiting a Windows bug that let it infect all computers connected to a remote printing network, thus searching for devices that were running Siemens’ software.

Zero-Day #1: LNK Files

The first of the four zero-day vulnerabilities Stuxnet exploited concerned LNK files, which are shortcuts used to refer to original file targets. The USBs used to spread Stuxnet installed LNK files containing links to a remotely hosted DLL file in the pathname that they used to store icon locations; because control panel links can have dynamic icons, the shell got the icon by running LoadLibrary() on the LNK target, automatically accessing the remotely hosted DLL file and executing DllMain(). Because the shell did not validate where the DLL was being loaded from, Stuxnet got code execution in the context of EXPLORER.exe, Windows’ file manager. While this attack did not provide elevation of privilege, it did provide the opportunity for rootkit installation via another bug exploit. Depending on whether the OS was Windows Vista and above or Windows XP and below, DllMain() either used Zero-Day #2 or #3 to obtain root access.

Zero-Day #2: Task Scheduler

The task scheduler used by Windows Vista and higher stored scheduled tasks in XML files with lax user privileges, allowing both reading and writing. DllMain() would create a task which the scheduler would checksum, hash, and save; after this, DllMain() edited the XML file so the principal executor of the task would be LocalSystem rather than the current user. To force a hash collision after the editing, the XML file was padded and hashed until its hash matched the original one. This collision was so easy to create because the task scheduler used the CRC32 algorithm, a notoriously weak one. After the edited XML file was hashed and saved, the task would eventually be executed by the LocalSystem, which, due to its high security clearance, would be allowed to install two rootkits onto the device.

Zero-Day #3: Keyboard Layout

Alternatively, in Windows XP and lower, Stuxnet would find the base memory address of the array used to store function pointers for loading various keyboard layouts; it would traverse the array until it found a memory address capable of storing a DWORD (a datatype used to store unsigned integers on Windows) in usermode. It would then allocate memory to that address and assign a function pointer to the rootkit installation code. The system call to load keyboards was done from kernel mode and without checking whether the integer provided was out of bounds, meaning that DllMain() called for a nonexistent keyboard layout and was instead provided rootkit installation.

Zero-Day #4: Spooler Subsystem

To spread, Stuxnet exploited a vulnerability in remote printer spooler RPCs, which are remote-procedure calls (RPCs). The print spooler would register RPC endpoints for printing protocols, where routines would create and execute MOF and EXE files. However, because Guests did not have access to file creation or execution, when Guest accounts on computers tried to print something on the remote printer, the RPCs would execute the request by impersonating a system user. Thus, Stuxnet would drop MOF and EXE files onto the printer, and from there onto other networked devices, by simply using guest-mode.

Bug #5: Input Process Images

To fake legitimate data when Stuxnet launched its attack, it exploited a naïve programming error: the process images were read/write, meaning it could simply overwrite the data being sent from the logic controllers to the SCADA and back.

Evidence Towards Cyberwarfare

Apart from the sophistication of Stuxnet, evidence towards its being a government project specifically targeting Iranian facilities is embedded in the data structures it used, DB 8063 and 8061, which utilize 164 element arrays, two-dimensional arrays with length 15 axes, and commonly repeat the number 6. The head of Iran’s nuclear program leaked that Natanz contained 164 centrifuges grouped into 15 stages of enrichment; on top of that, 6 is considered a special number in Iranian culture and was utilized in Natanz construction. In fact, the feed stage of enrichment was stage 6 of 15. In addition, the code made repeated use of 315, which happens to be the drive speed necessary for maximum rotor fatigue.

Aftermath

As one of the first cyberweapons employed by a national government, Stuxnet is seen as the beginning of the modern age of cyberwarfare, the precedent to China’s GhostNet and Russian election interference in 2016; the absence of a ‘kill’ command in Stuxnet enabled its eventual deconstruction and analysis by security firms, begetting an open source version that propagated among criminal groups and would be modified for dozens of high-profile attacks, including Duqu, Flame, and and Gauss, employed not only on the international level, but also for industrial sabotage and espionage. The greatest legacy of Stuxnet, however, was not the precedent it set, but the lessons it taught to the cybersecurity community regarding basic patching — three of the four zero-days Stuxnet exploited were easily patched — in fact, the Microsoft team that dissected Stuxnet claims most bugs were fixed within the hour they were discovered. Stuxnet sent a clear message it was basic protection that was integral to cybersecurity, not high-level threats.
